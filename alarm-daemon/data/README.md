# `data/` — distro packaging assets

This directory holds the system-bus / Yocto deployment files. They are not
used by `setup.sh` (which is the developer / CI installer and stays
session-bus + `systemd --user`); a Yocto recipe — or any Linux distro
packager — installs them verbatim.

## Layout

```text
data/
├── systemd/alarm-daemon.service           → /lib/systemd/system/alarm-daemon.service
├── dbus/org.helm.AlarmDaemon.conf         → /usr/share/dbus-1/system.d/org.helm.AlarmDaemon.conf
├── dbus/org.helm.AlarmDaemon.service      → /usr/share/dbus-1/system-services/org.helm.AlarmDaemon.service
├── dbus/org.helm.AlarmDaemon.Control.xml  → /usr/share/dbus-1/interfaces/org.helm.AlarmDaemon.Control.xml
├── sysusers.d/helm-alarm-daemon.conf      → /usr/lib/sysusers.d/helm-alarm-daemon.conf
├── tmpfiles.d/helm-alarm-daemon.conf      → /usr/lib/tmpfiles.d/helm-alarm-daemon.conf
└── etc/alarm-daemon.conf.example          → /etc/helm/alarm-daemon.conf  (rename, optional)
```

The binary itself lands at `/usr/bin/alarm-daemon`, and the bundled WAV
sounds at `/usr/share/helm/sounds/*.wav`.

## Yocto recipe sketch

A `cargo-bitbake`-generated recipe needs roughly the following installation
hook on top of the autogenerated build steps:

```bitbake
SUMMARY  = "Helm alarm daemon"
LICENSE  = "MIT | Apache-2.0"
LIC_FILES_CHKSUM = "file://LICENSE-MIT;md5=...   file://LICENSE-APACHE;md5=..."

inherit cargo cargo-update-recipe-crates systemd useradd

SYSTEMD_SERVICE:${PN} = "alarm-daemon.service"
USERADD_PACKAGES = "${PN}"
GROUPADD_PARAM:${PN} = "--system helm"
USERADD_PARAM:${PN}  = "--system --gid helm \
                        --home-dir /var/lib/helm --shell /usr/sbin/nologin \
                        --comment 'Helm alarm daemon' helm-alarm"

DEPENDS  += "alsa-lib"
RDEPENDS:${PN} += "alsa-lib dbus"

do_install:append() {
    install -D -m 0755 ${B}/release/alarm-daemon              ${D}${bindir}/alarm-daemon

    # Bundled sounds.
    install -d ${D}${datadir}/helm/sounds
    install -m 0644 ${S}/assets/alarms/*.wav                  ${D}${datadir}/helm/sounds/

    # systemd unit + dbus policy + activation file + introspection schema.
    install -D -m 0644 ${S}/data/systemd/alarm-daemon.service \
        ${D}${systemd_unitdir}/system/alarm-daemon.service
    install -D -m 0644 ${S}/data/dbus/org.helm.AlarmDaemon.conf \
        ${D}${datadir}/dbus-1/system.d/org.helm.AlarmDaemon.conf
    install -D -m 0644 ${S}/data/dbus/org.helm.AlarmDaemon.service \
        ${D}${datadir}/dbus-1/system-services/org.helm.AlarmDaemon.service
    install -D -m 0644 ${S}/data/dbus/org.helm.AlarmDaemon.Control.xml \
        ${D}${datadir}/dbus-1/interfaces/org.helm.AlarmDaemon.Control.xml

    # sysusers.d / tmpfiles.d (handled by the matching systemd helpers).
    install -D -m 0644 ${S}/data/sysusers.d/helm-alarm-daemon.conf \
        ${D}${sysconfdir}/sysusers.d/helm-alarm-daemon.conf
    install -D -m 0644 ${S}/data/tmpfiles.d/helm-alarm-daemon.conf \
        ${D}${sysconfdir}/tmpfiles.d/helm-alarm-daemon.conf
}

FILES:${PN} += " \
    ${systemd_unitdir}/system/alarm-daemon.service \
    ${datadir}/dbus-1/system.d/org.helm.AlarmDaemon.conf \
    ${datadir}/dbus-1/system-services/org.helm.AlarmDaemon.service \
    ${datadir}/dbus-1/interfaces/org.helm.AlarmDaemon.Control.xml \
    ${sysconfdir}/sysusers.d/helm-alarm-daemon.conf \
    ${sysconfdir}/tmpfiles.d/helm-alarm-daemon.conf \
    ${datadir}/helm/sounds/*.wav \
"
```

If your layer doesn't use `useradd.bbclass`, the `sysusers.d` file is enough
on systems where `systemd-sysusers.service` runs at first boot.

## D-Bus introspection schema

`dbus/org.helm.AlarmDaemon.Control.xml` is the **authoritative contract** for
the Control interface. The Rust `#[interface]` impl in `src/ipc/control.rs`
stays in sync via the integration test
`tests/introspect_xml.rs`, which boots the daemon, fetches its live
`Introspect()` reply, and asserts structural equivalence (method/signal/arg
identity, ignoring doc annotations).

Distro packagers should install the file at the freedesktop convention path:

```text
/usr/share/dbus-1/interfaces/org.helm.AlarmDaemon.Control.xml
```

Downstream consumers can:

- generate typed clients without a running daemon — e.g.
  `zbus-xmlgen file org.helm.AlarmDaemon.Control.xml`,
  `gdbus-codegen --c-namespace Helm`, or `dbus-fast` (Python).
- run API-stability checks across releases with `dbus-deviation`.
- render documentation: the file uses the `<doc:doc>` namespace from the
  freedesktop introspection DTD, picked up by `dbus-doc` and friends.

## Renaming the user / group

The `helm-alarm` user and `helm` group names are the project defaults and are
referenced from:

- `data/systemd/alarm-daemon.service` (`User=` / `Group=`)
- `data/dbus/org.helm.AlarmDaemon.conf` (`policy user=` / `policy group=`)
- `data/sysusers.d/helm-alarm-daemon.conf`
- `data/tmpfiles.d/helm-alarm-daemon.conf`

A vendor layer that needs different names should `sed`-patch all four files
(or override them with `bbappend` files); nothing in the binary hardcodes
either name.

## Verifying the install on a Yocto image

```bash
# As any user:
busctl --system list | grep org.helm.AlarmDaemon
busctl --system call \
    org.helm.AlarmDaemon /org/helm/AlarmDaemon \
    org.helm.AlarmDaemon.Control Status

# As a member of the `helm` group:
busctl --system call \
    org.helm.AlarmDaemon /org/helm/AlarmDaemon \
    org.helm.AlarmDaemon.Control Arm s "builtin:collision"
```

`systemd-analyze security alarm-daemon.service` should report a low exposure
score (< 2.0) thanks to the bundled hardening directives.

## Liveness supervision

The unit is `Type=notify` with `WatchdogSec=10s`; the daemon sends `READY=1`
after the D-Bus name is owned and pings `WATCHDOG=1` every 5s on the Tokio
runtime. A wedged executor (deadlock, blocked task) stops the pings, systemd
kills the process with `SIGABRT`, and `Restart=on-failure` brings it back up
within `RestartSec=2s`. Inspect with:

```sh
journalctl -u alarm-daemon.service -b | grep -i watchdog
systemctl show alarm-daemon.service -p WatchdogUSec,WatchdogTimestamp
```

To raise/lower the supervision window for a specific image, override the
unit instead of editing the upstream file:

```sh
mkdir -p /etc/systemd/system/alarm-daemon.service.d
cat >/etc/systemd/system/alarm-daemon.service.d/watchdog.conf <<'EOF'
[Service]
WatchdogSec=30s
EOF
systemctl daemon-reload && systemctl restart alarm-daemon.service
```
